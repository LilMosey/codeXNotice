# CodeX Notice 设计文档

## 目标

CodeX Notice 是一个 macOS 菜单栏常驻应用，用来在当前机器上的 Codex Desktop 长任务完成后通知用户。它主要解决用户离开电脑后，Codex 任务已经执行完成但用户收不到提醒的问题。

当前可打包 MVP 聚焦 macOS 上的 Codex Desktop 自动检测，以及通过 macOS 本地通知接入 Apple 生态。钉钉和延迟合并通知保留为后续阶段。

## 产品形态

应用以轻量菜单栏 App 的形式运行。用户打开 CodeX Notice 后，它会自动监听当前 macOS 用户下的 Codex Desktop 本地任务状态，并根据用户配置的通知规则发送通知。

用户安装后不需要额外安装 Node.js、Rust、Python 或其他运行环境。开发依赖只存在于源码构建阶段。

## 技术选型

第一版使用 Tauri、Rust 和 Web UI。

Rust 负责后台能力：

- Codex 任务检测
- 规则匹配
- SQLite 本地存储
- 通知发送
- 本地诊断

Web UI 负责配置和查看：

- 通知渠道配置
- 规则编辑和拖动排序
- 任务历史和通知历史
- 诊断信息

选择 Tauri 而不是 Electron 的原因是：这个应用是常驻型小工具，Tauri 的包体和后台资源占用更低，同时仍然可以用 Web 技术快速构建配置界面。

## 当前 MVP 范围

当前可打包 MVP 包含：

- macOS 菜单栏应用
- App 运行时自动检测当前机器上的 Codex Desktop 任务
- macOS 本地通知
- 可拖动排序的通知规则列表
- 最多 50 条通知规则
- 基于任务耗时的规则匹配
- 基于星期和时间段的通知时间控制
- 窗口外默认丢弃
- 任务历史和通知历史
- Codex 数据源可用性诊断
- 打包为 `.app` 和 `.dmg`

当前 MVP 不包含：

- 钉钉机器人 Webhook 通知
- 钉钉可选加签 Secret
- 延迟通知合并
- APNs 集成
- Apple Watch 独立直推
- 飞书
- 企业微信
- 微信
- 云端同步
- 账号体系或登录
- 远程服务器任务检测
- GitHub Actions 任务检测
- 其他机器上运行的 Codex 任务检测

当前 MVP 中的 Apple Watch 支持，是指通过 macOS 本地通知进入用户自己的 Apple 通知生态。是否能到达 Apple Watch，取决于用户系统和设备的通知配置。

## 任务检测

第一版的主路径是自动检测任务，不要求用户每次用包装命令启动 Codex。

CodeX Notice 会读取当前 macOS 用户下的 Codex Desktop 本地数据。第一优先级检测适配器读取 Codex 状态 SQLite 文件，例如：

- `~/.codex/state_*.sqlite`

该适配器优先寻找能够表达任务状态的字段，例如：

- 任务 id
- 任务名称或指令摘要
- 状态
- 创建时间
- 更新时间
- 开始时间
- 完成时间
- 最近错误

在当前参考机器上，`agent_jobs` 表包含 `started_at`、`completed_at`、`status`、`name`、`last_error` 等字段，因此它是第一版优先支持的任务完成检测来源。

应用还会提供 session 文件兜底适配器，读取：

- `~/.codex/sessions/**/rollout-*.jsonl`

当主 SQLite 数据源不可用、不完整，或 Codex 本地结构变化时，session 文件适配器作为兼容方案。它不如状态数据库精确，因此只作为兜底能力。

检测器会在 CodeX Notice 自己的数据库中维护已处理任务指纹，避免轮询和应用重启后重复通知。

任务检测只覆盖当前机器上的 Codex Desktop 任务。它不承诺检测其他机器、远程服务器或云端任务。

## 可选包装命令

第一版可以保留一个辅助命令作为备用和诊断入口：

```bash
codex-notice run -- <command>
```

它不是主要用户路径。它用于测试通知行为，也为后续 Codex CLI 或非桌面任务支持留接口。

## 规则模型

通知规则按优先级从上到下排列，用户可以拖动排序。

应用最多支持 50 条规则。

每个 Codex 任务完成后，规则引擎从第一条启用规则开始向下扫描。第一条“耗时条件”命中的规则会被选中。一旦选中规则，后面的低优先级规则不再参与判断。

这意味着：低优先级的“所有耗时、任何时间都通知”规则，不能覆盖高优先级窄规则的通知时间限制。

每条规则包含：

- 名称
- 启用或禁用
- 耗时条件
- 通知时间窗口
- 窗口外处理方式
- 通知渠道

耗时条件支持：

- 所有耗时
- 一个或多个闭区间，例如 `1-50 分钟`
- 开放下界区间，例如 `120 分钟以上`

内部统一用秒保存耗时。

通知时间窗口支持：

- 每时每刻
- 一个或多个“星期 + 时间段”
- 跨天时间段，例如 `22:00-02:00`

当前 MVP 的窗口外处理方式固定为丢弃。后续阶段再恢复每条规则单独配置丢弃或延迟。

首次启动时，如果用户还没有创建规则，应用自动创建一条默认规则：

- 耗时条件：所有耗时
- 通知时间：每时每刻
- 窗口外处理：丢弃
- macOS 本地通知：开启

## 规则示例

假设用户从上到下配置了三条规则：

1. 耗时 `1-50 分钟` 或 `120 分钟以上`；通知时间为周一到周五 `08:00-20:00`；窗口外处理为延迟。
2. 耗时 `60-70 分钟`；通知时间为周六 `10:00-11:00`。
3. 所有耗时；每时每刻通知。

如果一个任务耗时 30 分钟，并且在周六完成，它会先命中规则 1 的耗时条件。因为规则 1 当前不在通知时间窗口内，所以只执行规则 1 的窗口外处理。规则 3 虽然是“所有耗时、每时每刻通知”，但不会再参与判断。

如果规则 1 的窗口外处理是延迟，那么这个周六完成的任务会进入规则 1 的合并队列，并在下一次允许通知的时间点发送，例如周一 `08:00`。

## 通知时机

任务完成后的判断流程：

1. 检测器记录任务。
2. 规则引擎找到第一条耗时命中的规则。
3. 如果没有规则命中，任务记录为忽略，不发送通知。
4. 如果命中的规则允许在任务完成时间通知，立即发送通知。
5. 如果命中的规则不允许在任务完成时间通知：
   - 丢弃策略：记录为已丢弃，不发送通知。
   - 延迟策略：加入该规则的待合并通知批次。

## 后续阶段：延迟合并通知

延迟通知不进入当前可打包 MVP。后续阶段恢复后，延迟通知按规则合并。

当多个任务在某条规则的通知时间窗口外完成时，它们会进入这条规则的待合并批次。到下一次允许通知的时间点时，应用只为这条规则发送一条合并通知。

合并通知包含：

- 规则名称
- 完成任务数量
- 成功数量
- 失败数量
- 最长耗时
- 完成时间范围
- 当任务数不超过 10 个时展示任务摘要

当批次内任务超过 10 个时，通知只展示统计信息。完整明细在 History 页面查看。

每个延迟任务只会进入一次合并通知。发送成功后，批次标记为已发送，不重复通知。

延迟任务有保留时长限制，默认 24 小时。超过保留时长的延迟任务标记为过期，不再发送。

## 通知渠道

当前 MVP 只支持 macOS 本地通知。钉钉保留为后续阶段。

### 后续阶段：钉钉

钉钉渠道后续使用机器人 Webhook。

配置项：

- 是否启用
- Webhook 地址
- 可选加签 Secret
- 测试通知

Secret 只保存在本地。UI 保存后不应明文展示 Secret。

### macOS 本地通知

Apple 生态通知使用 macOS 本地通知。

配置项：

- 是否启用
- 测试通知

这是第一版的 Apple 生态集成方式。Apple Watch 是否收到提醒，取决于用户系统通知设置。

## 通知内容

单任务通知包含：

- 应用名称
- 规则名称
- 任务名称或摘要
- 成功或失败状态
- 耗时
- 完成时间
- 可用时展示错误摘要

合并通知包含：

- 应用名称
- 规则名称
- 任务数量
- 成功和失败数量
- 最长耗时
- 完成时间范围
- 任务数不超过 10 个时展示任务摘要

## 菜单栏 UI

菜单栏包含：

- 打开 CodeX Notice
- 最近任务状态
- 暂停通知
- 恢复通知
- 发送测试通知
- 退出

暂停和恢复只影响通知发送，不影响任务检测。暂停期间检测到的任务仍然记录到 History，但不立即发送。这个行为需要在 History 中清晰展示。

## 配置 UI

主窗口包含四个区域。

### Rules

Rules 区域支持：

- 最多 50 条规则
- 拖动排序
- 启用或禁用规则
- 编辑名称
- 编辑耗时条件
- 编辑星期和时间窗口
- 编辑窗口外处理方式
- 选择通知渠道

### Channels

Channels 区域支持：

- 钉钉配置
- 钉钉测试发送
- macOS 本地通知测试
- 渠道健康状态显示

### History

History 区域展示：

- 检测到的任务
- 命中的规则
- 通知决策
- 通知状态
- 延迟合并批次详情
- 发送失败原因

### Diagnostics

Diagnostics 区域展示：

- 检测到的 Codex 数据路径
- 当前启用的检测适配器
- 最近一次成功扫描时间
- 扫描错误
- 权限问题
- Codex schema 兼容性提示
- 应用版本

## 本地存储

CodeX Notice 使用自己的 SQLite 数据库。

数据库保存：

- 应用设置
- 通知渠道
- 规则和规则顺序
- 检测到的任务
- 通知事件
- 延迟批次
- 适配器扫描状态
- 诊断日志

应用不会写入 Codex 自己的数据文件，只读取 Codex 本地状态。

## 核心 Rust 模块

`detection` 负责发现已完成的 Codex 任务，并归一化为应用内部任务记录。

`rules` 负责按顺序评估规则，并返回唯一的通知决策。

`scheduler` 负责管理延迟合并批次，并在规则进入可通知时间时发送。

`notifications` 负责在统一接口后面发送钉钉和 macOS 本地通知。

`db` 负责 SQLite schema 和迁移。

`diagnostics` 负责向 UI 报告适配器健康状态和最近失败。

`config` 负责读取和保存用户配置。

`cli` 负责可选包装命令和诊断入口。

## 风险

主要产品风险是 Codex Desktop 本地状态不是公开稳定 API。表、字段或文件路径可能变化。

缓解方式：

- 将 Codex 读取逻辑隔离在检测适配器后面
- 永远不写入 Codex 数据文件
- 在 Diagnostics 中展示适配器健康状态
- 扫描状态保存在 CodeX Notice 自己的数据库
- 为已知 Codex schema 提供 fixture 测试
- 保留包装命令作为兜底和诊断入口

另一个风险是通知过多。优先级规则、每条规则独立窗口外策略、暂停控制和合并通知机制，都是为了让通知保持有用而不是制造打扰。
